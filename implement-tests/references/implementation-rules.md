# テスト実装ルール詳細

## 1. テストデータのライフサイクル管理（必須）

テストデータの作成と削除を同一スコープに配置し、テストの完全な独立性を保証する。

### 原則

- テストデータの作成は、前提条件を共有する `describe` ブロックの `beforeEach` で行う
- テストデータの削除（クリーンアップ）は、データを作成した `beforeEach` 内でテストフレームワークのフック機能を利用して登録する
- これにより、テストの成否に関わらず確実に状態がリセットされる

### 実装イメージ（Vitest の場合）

```typescript
beforeEach(async () => {
  // データ作成
  await createTestData();

  onTestFinished(async () => {
    // データ削除（テスト成否に関わらず実行される）
    await cleanupTestData();
  });
});
```

### 実装イメージ（Jest の場合）

```typescript
beforeEach(async () => {
  // データ作成
  await createTestData();
});

afterEach(async () => {
  // データ削除
  await cleanupTestData();
});
```

### ポイント

- セットアップとティアダウンの凝集度を高く保つ
- ネストされた `describe` では、そのスコープ固有のデータのみを管理する
- グローバルな状態変更は避ける

## 2. AAA パターンの徹底

各テストケース内を空行で視覚的に3フェーズに分け、役割を混同させない。

### Arrange（準備）

- テスト固有の前提条件を構築する
- 共通の前提条件は `beforeEach` に配置する
- API モックの定義やテストデータの追加セットアップを行う

### Act（実行）

- テスト対象の振る舞いを **1つだけ** 実行する
- `it` ブロック内で実行する

### Assert（検証）

- 事後条件と不変条件を検証する
- `it` ブロック内で実行する

### 例

```typescript
it("在庫不足の場合は購入失敗し在庫は変わらない", async () => {
  // Arrange
  const initialStock = await getStock(productId);

  // Act
  const result = await purchase(productId, initialStock + 1);

  // Assert — 事後条件
  expect(result.success).toBe(false);
  expect(result.error).toBe("INSUFFICIENT_STOCK");
  // Assert — 不変条件
  const currentStock = await getStock(productId);
  expect(currentStock).toBe(initialStock);
});
```

## 3. モックの最小化

### 禁止事項

- テスト対象の内部モジュール（Repository、Service、API Client 等）に対する `vi.mock` / `jest.mock` / `vi.spyOn`
- 呼び出し回数の検証（`toHaveBeenCalledTimes` 等）を主たるアサーションにすること

### 許可されるモック

| 対象 | 方針 |
|---|---|
| 外部 HTTP API | HTTP レイヤーでのインターセプト（msw、nock 等）またはプロジェクトのモックサーバーを使用 |
| データベース | プロジェクトの慣習に従う（テスト用DB、インメモリDB、またはモックサーバー） |
| ファイルシステム・時刻等 | 必要に応じてスタブ化を許可 |

### 判断基準

プロジェクトの既存テストを確認し、確立されたモック戦略に従う。既存の慣習がない場合は、外部境界のみモックする古典派アプローチを採用する。

## 4. 契約による設計の検証

### 事後条件の検証（正常系）

Act フェーズで得られた結果を多角的に検証する：

- **戻り値（出力値）**: 関数の返り値が期待通りか
- **状態変化**: データベースやストアの更新後の状態が期待通りか（SELECT して確認）
- **副作用**: 必要に応じて、イベント発火やログ出力等を確認

### 不変条件の検証（異常系）

エラー発生時は、エラー自体の検証に加えて「状態が変わっていないこと」を必ず確認する：

- エラーがスローされること
- データベースの状態が Act 実行前と変わっていないこと（中途半端な更新がないこと）
- 関連する状態（カウンター、残高等）が変化していないこと

## 5. 可読性と保守性

- `beforeEach` にデータ準備をまとめる場合でも、各 `it` を読むだけで「何がテストされているか」「なぜそのアサートになるのか」が推測できるようにする
- テストケース名はビジネス上の意味を伝える（メソッド名を含めない）
- マジックナンバーは避け、意味のある変数名をつける

## 6. テストユーティリティの活用

- プロジェクトの `test-utils/` や `__helpers__/` 等に既存のユーティリティがあれば積極的に使う
- 繰り返しパターンがあれば、新しいユーティリティ関数として抽出する
- テストコードが増えるにつれ、ユーティリティの改善が必要ならゼロベースで再構築する
