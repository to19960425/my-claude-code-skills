# テストアウトライン実装ガイドライン

テストアウトライン（`it.todo` を含むファイル）を実行可能なテストコードに変換する際のルールと手順。

## 出力要件

1. 提供されたアウトラインの `describe` ブロック構造とテストケース名（ドキュメントとしての意図）をそのまま維持する
2. すべての `it.todo` を実行可能な `it` に変更し、テストコードを実装する
3. 実装完了後、テストを実行して結果を確認する
   - テストコードが間違っている場合 → テストコードを修正する
   - プロダクションコードやテスト内容自体が間違っている場合 → 該当テストをスキップ（`it.skip`）してユーザーに確認する

## ワークフロー

```
1. アウトラインを読み、テスト対象コードも読む
2. プロジェクトの既存テストからテストフレームワーク・慣習を把握する
3. テストユーティリティ（test-utils 等）が存在すれば把握・活用する
4. it.todo を順に実装する
5. テストを実行し、失敗があれば判断・修正する
6. 全テスト完了を確認する
```

## 絶対のルール

詳細は [references/implementation-rules.md](references/implementation-rules.md) を参照。

1. **テストデータのライフサイクル管理** — セットアップとティアダウンを同じスコープに配置し、テストの独立性を保証する
2. **AAA パターンの徹底** — Arrange / Act / Assert を視覚的に分離する
3. **モックの最小化** — `vi.mock` / `jest.mock` によるテスト対象内部モジュールのモック化を禁止する。モックは外部境界のみ
4. **契約による設計の検証** — 事後条件（戻り値 + 状態変化）と不変条件（異常系で状態が変わっていないこと）を検証する
5. **可読性** — 各 `it` を読むだけで意図がわかるようにする
6. **テストユーティリティの活用** — 既存のユーティリティを使い、必要なら追加する
