# テストケーススケルトン生成

## 概要

ユーザーから提示された機能仕様・要件に基づき、テストケースの階層構造（describe + it.todo）のみを生成する。実装コードやテストロジックは一切書かない。

## ワークフロー

### 1. 入力の確認

ユーザーから以下の情報を受け取る。不足している場合は質問して補完する。

- **実現したいこと**: 機能の概要・目的
- **制約**: ビジネスルール、数値制限、デフォルト値など
- **対象のコード**: 既存コードがある場合はファイルパスや関数名

### 2. 仕様の理解と不明点の解消

- 提示された仕様を分析し、データフローや状態遷移を把握する
- 不明瞭な点、曖昧な要件、暗黙の前提がある場合は**必ずユーザーに質問する**
- 推測で補完しない。確認してから進める

### 3. テストケースの設計

以下の観点でテストケースを網羅的に洗い出す:

- **正常系**: 基本的な成功パス
- **境界値**: 上限・下限・ちょうどの値
- **異常系**: 無効な入力、エラー条件
- **エッジケース**: 空データ、null/undefined、型の不一致
- **制約条件**: 仕様で明示された制約の検証

### 4. テストファイルの生成

テストケースをファイルに出力する。

## 出力ルール

### 必須

- テストランナーは **プロジェクトで使用しているもの** を自動検出して使う（Vitest, Jest等）
- `describe` と `it.todo` のみを使用する
- `it.todo` の引数（テストケース名）は日本語で「〜の時、〜であること」の形式にする
- 各 `it.todo` の直前に、そのテストの意図を示すコメントを添える
- テストファイルの配置はプロジェクトの規約に従う

### 禁止

- テストロジック（arrange, act, assert）を書かない
- モック・スタブ・スパイの設定を書かない
- import文以外の実行コードを書かない
- モック利用を前提としたテストケースを作らない

### テストケース名の例

```
// 良い例
it.todo('税率10%で1000円の商品の場合、合計が1100円であること')
it.todo('在庫が0の場合、購入に失敗しエラーメッセージが返ること')
it.todo('検索結果が空の場合、空配列が返ること')

// 悪い例
it.todo('calculateTaxのテスト')
it.todo('正常系')
it.todo('エラーハンドリング')
```

## 出力フォーマット例

```typescript
import { describe, it } from 'vitest'; // プロジェクトに合わせて変更

describe('[機能名]', () => {
  describe('正常系', () => {
    // 基本的な成功パスの検証
    it.todo('[条件]の時、[期待結果]であること');
  });

  describe('境界値', () => {
    // 上限・下限値での振る舞いの検証
    it.todo('[境界条件]の時、[期待結果]であること');
  });

  describe('異常系', () => {
    // エラー条件での振る舞いの検証
    it.todo('[異常条件]の時、[期待結果]であること');
  });
});
```
