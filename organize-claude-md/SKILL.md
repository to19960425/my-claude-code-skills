# CLAUDE.md 整理コマンド

CLAUDE.mdをベストプラクティスに沿って整理する。以下の手順を順番に実行すること。

## 手順1: 現状分析

プロジェクトルートの`CLAUDE.md`を読み、以下を把握する:
- 現在の行数
- セクション構成
- `.claude/rules/`配下の既存ファイル一覧

分析結果をユーザーに報告してから次へ進む。

## 手順2: 削除テスト

各行・各ブロックに「この行を削除したらClaudeはミスするか？」を適用する。

**削除すべき情報:**
- コードを読めばわかること（エンティティ関係、関数シグネチャ、内部実装の詳細）
- 言語やフレームワークの標準的慣例
- ESLint・PHPCSなどツールに委譲できるスタイルルール
- 長い説明やチュートリアル的な記述
- 詳細なAPIドキュメント（リンク参照で代替）
- 頻繁に変わる情報（Auto Memoryに任せるべき内容）

**残すべき情報:**
- プロジェクト固有のコマンド（コードから読み取れない）
- デフォルトと異なるスタイルや規約
- テスト戦略・優先順位
- アーキテクチャ上の設計判断・意図
- 過去の失敗に基づく非自明な注意事項（落とし穴）

削除候補の一覧をユーザーに提示し、確認を取ってから次へ進む。

## 手順3: 5セクション構成に再編

以下の構成に再編する:

```
# CLAUDE.md

## プロジェクト概要
（1-2行。何のプロジェクトか、主要技術スタック）

## ガイドライン
（Claudeのデフォルト動作と異なるルールのみ。否定だけの指示は避け、代替手段を併記する）

## コマンド
（ビルド・テスト・リント・デプロイ等。コードブロックで記載）

## アーキテクチャ
（主要ディレクトリと役割。5-7項目程度）

## 注意事項
（プロジェクト固有の落とし穴。変更時に壊れやすい箇所や順序依存）
```

追加ルール:
- 全体で500行以内を目標とする
- 「IMPORTANT」等の強調は本当にミスが起きやすい箇所のみに使う
- セクション内は箇条書きで簡潔に書く

## 手順4: Progressive Disclosure

CLAUDE.mdから溢れる詳細情報を`.claude/rules/`に分離することを提案する。

- `.claude/rules/*.md`は関連するコンテキストでClaude Codeが自動的に読み込む
- 特定のエンティティや機能に関する詳細な注意事項はルールファイルに移動する
- 既に`.claude/rules/`にファイルがある場合は、重複を避けて統合する

分離案をユーザーに提示し、確認を取ってから次へ進む。

## 手順5: 書き出し

整理後のCLAUDE.mdを書き出す。変更サマリとして以下を報告する:
- 整理前後の行数比較
- 削除した項目の要約
- `.claude/rules/`に移動した項目（あれば）
- 残った各セクションの概要
